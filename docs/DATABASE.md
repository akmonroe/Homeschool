# Homeschool database structure

This document describes **where data lives**, **how the two databases relate**, and **what each table is for**. It is meant to stay accurate as a reference when changing the app or adding new features.

---

## 1. Executive summary

The Homeschool stack uses **two separate persistence layers**:

| Store | Engine | Location (typical) | Role |
|-------|--------|--------------------|------|
| **Core platform** | PostgreSQL 16 | `DATABASE_URL` → database `homeschool`, schema **`core`** | Canonical **students**, cross-app **projects**, **assignments**, **grades**, **skill observations** — shared by all suite apps. |
| **Dictation app** | SQLite 3 | File **`dictation.db`** under `DICTATION_DATA_DIR` (default `/app/data` in Docker) | Spelling **dictionary**, **per-student word queue**, **practice history**, and runtime paths for **TTS** — specific to the dictation feature. |

**Identity bridge:** Each dictation “user” row can store **`core_student_id`**, a **string copy** of the Postgres `core.students.id` (UUID). That links the integer dictation profile to the platform student. The canonical student record is always **`core.students`**; dictation rows are a **materialized profile** for the spelling engine.

## 2. PostgreSQL (`core` schema)

### 2.1 Connection and migrations

- **Runtime URL:** `DATABASE_URL`, typically `postgresql+asyncpg://...` for the FastAPI app (async SQLAlchemy).
- **Migrations:** [Alembic](https://alembic.sqlalchemy.org/). Revision scripts live in `alembic/versions/`. The **Alembic version table** (`alembic_version`) lives in the **`public`** schema so migrations can run before `core` exists.
- **Boot:** The Docker entrypoint runs `alembic upgrade head` before starting uvicorn (using a sync driver URL for Alembic).

### 2.2 Schema: `core`

All objects below are in **`core.*`** unless noted.

#### `core.students`

Canonical learner record for the whole suite.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | Primary key; generated by the app on insert. |
| `display_name` | `VARCHAR(255)` | Required. |
| `notes` | `TEXT` | Optional free text. |
| `metadata` | `JSONB` | Default `{}`. **Convention:** `dictation_skill_level` (integer 1–10) is read when syncing dictation and when drafting AI word lists from platform admin. |
| `created_at`, `updated_at` | `TIMESTAMPTZ` | Defaults / updates via DB or ORM. |

**Relationships:** Other `core` tables reference `students.id` with `ON DELETE CASCADE` where a child row cannot exist without the student.

---

#### `core.projects`

Optional **grouping** for work across apps (unit, portfolio, AI-generated path).

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | PK. |
| `student_id` | `UUID` | FK → `core.students.id`, CASCADE delete. |
| `title` | `VARCHAR(500)` | Required. |
| `description` | `TEXT` | Optional. |
| `status` | `VARCHAR(32)` | Default `active`. |
| `originating_app` | `VARCHAR(64)` | Optional slug (e.g. which app created it). |
| `metadata` | `JSONB` | Default `{}`. |
| `created_at`, `updated_at` | `TIMESTAMPTZ` | |

**Index:** `ix_core_projects_student_id` on `(student_id)`.

---

#### `core.assignments`

A **unit of assigned work** for one student. Intended for humans, UIs, and future **AI agents** (title, instructions, `rubric_json`, `metadata`).

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | PK. |
| `student_id` | `UUID` | FK → `students.id`, CASCADE. |
| `project_id` | `UUID` nullable | FK → `projects.id`, **SET NULL** on project delete. |
| `title` | `VARCHAR(500)` | Required. |
| `app_slug` | `VARCHAR(64)` nullable | e.g. `dictation` when created from the spelling word planner. |
| `status` | `VARCHAR(32)` | Default `draft`. |
| `due_at` | `TIMESTAMPTZ` nullable | |
| `instructions` | `TEXT` nullable | |
| `rubric_json` | `JSONB` nullable | Structured rubric for scoring / agents. |
| `metadata` | `JSONB` | Default `{}`. Example: `dictation_user_id`, `word_count` when committing AI word lists. |
| `created_at`, `updated_at` | `TIMESTAMPTZ` | |

**Indexes:** `ix_core_assignments_student_id`, `ix_core_assignments_project_id`.

---

#### `core.assignment_items`

**“Tall” decomposition** of an assignment: many rows per assignment, ordered by `sequence`, each with a **`item_type`** and **`payload_json`**. Use this for steps, prompts, per-word lines, or agent checkpoints **without** widening `assignments`.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | PK. |
| `assignment_id` | `UUID` | FK → `assignments.id`, CASCADE delete. |
| `sequence` | `INTEGER` | Default `0`; order items for display / execution. |
| `item_type` | `VARCHAR(64)` | e.g. `spelling_word`. |
| `payload_json` | `JSONB` | Default `{}`. e.g. `{"word": "apple"}`. |
| `created_at` | `TIMESTAMPTZ` | |

**Index:** `ix_core_assignment_items_assignment_id` on `(assignment_id)`.

---

#### `core.grades`

A **graded outcome** (human or future agent). Optional links to assignment and/or project.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | PK. |
| `student_id` | `UUID` | FK → `students.id`, CASCADE. |
| `assignment_id` | `UUID` nullable | FK → `assignments.id`, SET NULL. |
| `project_id` | `UUID` nullable | FK → `projects.id`, SET NULL. |
| `scored_by` | `VARCHAR(32)` | Default `human`; room for `agent`, etc. |
| `score_numeric`, `score_max` | `NUMERIC(12,4)` nullable | |
| `letter` | `VARCHAR(8)` nullable | |
| `feedback` | `TEXT` nullable | |
| `rubric_scores_json` | `JSONB` nullable | |
| `evidence_refs_json` | `JSONB` nullable | Pointers to artifacts, attempts, files. |
| `metadata` | `JSONB` | Default `{}`. |
| `created_at` | `TIMESTAMPTZ` | |

**Indexes:** `ix_core_grades_student_id`, `ix_core_grades_assignment_id`.

---

#### `core.skill_observations`

**Append-friendly** skill signals over time (not only dictation). Query by `student_id` and optional `skill_key`.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `UUID` | PK. |
| `student_id` | `UUID` | FK → `students.id`, CASCADE. |
| `skill_key` | `VARCHAR(128)` | e.g. `spelling.level`. |
| `scale_min`, `scale_max`, `value_numeric` | `NUMERIC` nullable | |
| `value_text` | `TEXT` nullable | |
| `source` | `VARCHAR(32)` | Default `system`. |
| `confidence` | `NUMERIC(5,4)` nullable | 0–1 style. |
| `context_assignment_id` | `UUID` nullable | FK → `assignments.id`, SET NULL. |
| `observed_on` | `DATE` nullable | Logical observation date. |
| `metadata` | `JSONB` | Default `{}`. |
| `created_at` | `TIMESTAMPTZ` | |

**Indexes:** `ix_core_skill_obs_student_id`, `ix_core_skill_obs_student_skill` on `(student_id, skill_key)`.

### 2.3 ER-style relationships (Postgres)

```text
students (1) ──< (N) projects
students (1) ──< (N) assignments ──< (N) assignment_items
students (1) ──< (N) grades
students (1) ──< (N) skill_observations

projects (1) ──< (N) assignments   [project_id nullable]
projects (1) ──< (N) grades       [project_id nullable]

assignments (1) ──< (N) assignment_items
assignments (1) ──< (N) grades    [assignment_id nullable]
assignments (1) ──< (N) skill_observations [context_assignment_id nullable]
```

## 3. SQLite (dictation: `dictation.db`)

### 3.1 File and lifecycle

- **Path:** `os.path.join(DICTATION_DATA_DIR, "dictation.db")` — default **`/app/data/dictation.db`** in the container; persist **`dictation-data`** volume if you care about durability.
- **Initialization:** `app.apps.dictation.database.init_db()` creates tables if missing and ensures **`core_student_id`** exists on `users` (see below).
- **Every connection:** `get_db_connection()` runs a small migration check so **older files** on disk gain `core_student_id` without manual SQL.

### 3.2 Tables

#### `users`

Dictation-specific profile. **`name`** is unique in SQLite (display name used by the student UI).

| Column | Type | Notes |
|--------|------|--------|
| `id` | `INTEGER` | PK, autoincrement. **This is what the dictation API calls `user_id`.** |
| `name` | `TEXT` | UNIQUE NOT NULL. |
| `difficulty_level` | `TEXT` | Historically string; app uses numeric levels as string (e.g. `"5"`) for AI prompts. |
| `created_at` | timestamp | Default current time. |
| `core_student_id` | `TEXT` nullable | **Link to Postgres:** string form of `core.students.id` (UUID). Unique when not NULL via partial index `ix_users_core_student_id`. |

**Sync rule:** When a `core.students` row is created or updated from platform admin, code upserts/updates the dictation `users` row with matching `core_student_id` and aligned `name` / difficulty.

---

#### `words`

Global spelling dictionary (shared across dictation users).

| Column | Type | Notes |
|--------|------|--------|
| `id` | `INTEGER` | PK. |
| `word` | `TEXT` | UNIQUE NOT NULL, normalized lowercase in app logic. |
| `difficulty_level` | `TEXT` | |
| `definition` | `TEXT` nullable | |
| `created_at` | timestamp | |

---

#### `user_words`

Join table: which words are **assigned** to which dictation user, plus spaced-repetition fields.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `INTEGER` | PK. |
| `user_id` | `INTEGER` | FK → `users.id`. |
| `word_id` | `INTEGER` | FK → `words.id`. |
| `interval`, `correct_streak` | `INTEGER` | Scheduling / streak. |
| `next_review_date` | `DATE` | Due logic for “words due today”. |
| **Constraint** | | `UNIQUE(user_id, word_id)`. |

---

#### `history`

Per-attempt log for grading in the dictation app.

| Column | Type | Notes |
|--------|------|--------|
| `id` | `INTEGER` | PK. |
| `user_id` | `INTEGER` | FK → `users.id`. |
| `word_id` | `INTEGER` | FK → `words.id`. |
| `is_correct` | boolean | |
| `attempt_date` | `DATE` | |

This is **dictation-local**; it is **not** the same as `core.grades` (which is cross-app and richer). Long term you may mirror or migrate attempts into `core.grades` or `core.skill_observations`.

---

### 3.3 ER-style relationships (SQLite)

```text
users (1) ──< (N) user_words >── (N) words   [many-to-many via user_words]
users (1) ──< (N) history     >── words
```

## 4. Cross-database behavior (important)

1. **Source of truth for “who is the student?”**  
   **`core.students`** (Postgres). Display name, notes, and extensible **`metadata`** live there.

2. **Why dictation still has `users`**  
   The dictation feature was built first with integer IDs and SQLite-friendly queries. **`users`** is the **execution profile** for spelling: word queue, history, and compatibility with existing `/apps/dictation/*` routes.

3. **Link column**  
   **`users.core_student_id`** = `str(core.students.id)` (UUID string). At most **one** dictation user row per core student (enforced by partial unique index when the column is set).

4. **AI word assignment (platform admin)**  
   - Draft/commit calls Ollama and **writes words to SQLite** (`words`, `user_words`).  
   - Commit also creates **`core.assignments`** + **`core.assignment_items`** so the suite has a **portable record** of what was assigned, independent of dictation’s internal IDs.

5. **No distributed transaction**  
   Postgres and SQLite are **not** updated in a single 2PC transaction. Failures can theoretically leave one side updated; the API is designed to surface errors (e.g. core student saved but dictation sync failed) so operators can retry or fix data.

## 6. Operational notes

- **Backups:** Back up Postgres (`homeschool`) and the **`dictation-data`** volume (or `dictation.db`) **together** if you need a consistent story for linked students.
- **Resetting dictation only:** Replacing `dictation.db` loses queues and history; **`core.students`** remain unless you delete them in Postgres. Re-sync students via platform admin or your own script.
- **Adding new apps:** Prefer new Postgres tables under **`core`** or a new schema, with **`student_id` → core.students.id**; avoid introducing a second student identity store.
